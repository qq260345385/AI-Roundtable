# AI Roundtable Evaluation Notes

这份文档用于观察真实模型圆桌会议是否有效。它不是自动评分规则，而是人工评估时的检查清单。

## 如何判断一次会议是否有效

一次有效的真实模型会议，应该让人看出“不同模型在同一个议题上如何形成观点、互相补充、保留分歧，并沉淀为可用纪要”。

可以从这些问题开始判断：

- 会议是否真的帮助理解问题，而不是只是生成多段相似回答？
- 第二阶段是否利用了其他模型的观点？
- 最终小结是否保留了分歧，而不是把所有观点磨平成一个安全答案？
- 会议结果是否足够清楚，能被复制到 Markdown 后继续讨论？
- 如果出现“模型调用失败记录”，是否仍然保留了其他模型的有效发言，并给出了可操作的排查建议？

## 需要观察的现象

模型发言是否有差异：

- 不同模型是否有不同表达方式、关注重点或推理路径？
- 差异是否来自模型自身输出，而不是人为角色分工？

是否出现无意义附和：

- 第二阶段是否只是“我同意前面观点”？
- 是否有新的补充、限制条件、例子或替代方案？

是否保留分歧：

- 总结是否明确列出主要分歧？
- 分歧是否具体，而不是泛泛写“仍有不同意见”？

是否有少数派观点：

- 是否出现不占主流但有价值的提醒？
- 少数派观点是否被总结保留下来？

总结是否过度折中：

- 总结是否为了显得平衡而失去判断？
- 是否能指出更推荐的下一步，而不是只说“需要综合考虑”？

成本和耗时是否可接受：

- 一次会议用了几个模型、几轮调用？
- 等待时间是否能接受？
- 输出质量是否值得这次调用成本？

部分 provider 失败时是否可解释：

- 失败记录是否明确标出 provider、模型和阶段？
- 失败原因是否简短、安全，不包含 API key、Authorization header 或完整第三方错误体？
- 建议操作是否能帮助用户检查 API key、MODEL、base URL、网络、限流或额度？

## 推荐测试问题

- AI Roundtable 和普通聊天机器人最大的差异是什么？
- 多模型圆桌会议应该如何避免变成多个相似答案的堆叠？
- 如果真实模型之间观点冲突，产品应该如何展示分歧？
- 在不使用数据库的第一版中，会议结果应该如何保存和复用？
- 如何判断一个真实模型 provider 的输出质量足够接入产品？
- 什么时候应该保留少数派观点，什么时候应该收敛成共识？
- AI Roundtable 的第一批用户最可能在哪些场景中使用它？
